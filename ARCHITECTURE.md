
# 🏗 Архитектура сервиса и MLOps-концепты

## 1. Монолит vs Микросервисы: обоснование выбора
Для текущего учебного проекта выбран **монолитный подход**.  
**Преимущества для данного кейса:**
- Низкая задержка (latency) при инференсе: все компоненты (предобработка, модель, API) работают в одном процессе.
- Простота развёртывания и отладки: один Docker-контейнер, минимум сетевых взаимодействий.
- Достаточно для MVP и нагрузок до ~50-100 RPS.

**Переход к микросервисам (гипотетический сценарий масштабирования):**  
При росте нагрузки или необходимости независимого масштабирования компонентов архитектура будет разделена на:
- `api-gateway` (приём запросов, rate-limiting)
- `preprocessing-service` (нормализация, кодирование)
- `inference-service` (запуск модели, масштабируется горизонтально)
- `logging-service` (асинхронная запись предсказаний)

## 2. Концепт асинхронной обработки (RabbitMQ)
В production-среде синхронный REST-API может стать узким местом при пакетной обработке заявок (batch scoring) или при высокой конкурентности.  
**Решение:** внедрение брокера сообщений **RabbitMQ** по паттерну `Publisher → Queue → Consumer`.
- **Сценарий:** Frontend/CRM кладёт заявку в очередь `credit_requests`. Воркер `inference-service` забирает сообщения, прогоняет через модель и отправляет результат в очередь `credit_results`.
- **Преимущества:** отказоустойчивость, буферизация пиковых нагрузок, возможность перезапуска воркеров без потери данных.

## 3. Логирование и мониторинг
В коде API реализовано **JSON-логирование** запросов и ответов (`logging` с кастомным форматом).  
**Production-интеграция:** Логи собираются агентом (Filebeat/Fluentd) и отправляются в **ELK-стек**:
- `Elasticsearch` — хранение и полнотекстовый поиск логов.
- `Logstash` — парсинг, фильтрация, обогащение данных.
- `Kibana` — визуализация дашбордов (ошибки 5xx, latency, распределение предсказаний).

## 4. Инструменты MLOps (DVC и MLflow)
- **DVC (Data Version Control):** Версионирование датасетов и артефактов моделей. В Git хранятся только `.dvc`-файлы с хешами, сами данные лежат в S3/Google Drive. Гарантирует воспроизводимость пайплайна при изменении входных данных.
- **MLflow Tracking:** Фиксация гиперпараметров, метрик (F1, Precision, Recall), версий зависимостей и путей к артефактам для каждого эксперимента. Позволяет сравнивать запуски и автоматически регистрировать лучшую модель в `Model Registry`.

## 5. Оптимизация инференса: uWSGI + NGINX и ONNX
- **uWSGI + NGINX:** Встроенный сервер Flask однопоточный и ограничен GIL. Связка `NGINX (reverse proxy + static) → uWSGI (multiprocess WSGI)` обеспечивает многопоточную обработку запросов, балансировку нагрузки и защиту от медленных клиентов.
- **ONNX-ML:** Для снижения latency и кросс-платформенного деплоя модель можно конвертировать в формат ONNX через `sklearn-onnx`. Это позволяет запускать инференс на C++/Java-серверах или через `ONNX Runtime` с аппаратным ускорением (GPU/CPU), минуя overhead Python.

## 6. Бизнес-метрики
Помимо технических метрик, для заказчика важны:
1. **Ожидаемые финансовые потери (Expected Loss Reduction):** `EL = PD × LGD × EAD`. Модель снижает `PD` (Probability of Default) за счёт точного отсечения рискованных заявок.
2. **Approval Rate при фиксированном уровне риска:** Доля одобренных заявок, при которой доля просрочек > 90 дней не превышает целевой порог (например, 3%). Позволяет балансировать между прибыльностью и риск-аппетитом банка.
