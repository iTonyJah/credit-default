# 🧪 План A/B-тестирования моделей скоринга

## 1. Цель и гипотеза
**Цель:** Оценить эффективность новой версии модели (`v2`, например, `GradientBoosting`) относительно текущей baseline-модели (`v1`, `RandomForest`) в реальных условиях продакшена.  
**Гипотеза:** Внедрение модели `v2` увеличит `F1-score` класса дефолта на ≥ 5% относительно `v1` без ухудшения `Precision`.

## 2. Дизайн эксперимента
- **Разделение трафика:** Случайное распределение 50/50 на уровне `user_id` или `session_id` через reverse-proxy/feature-flag.
- **Изоляция:** Пользователь попадает только в одну группу на весь период теста (исключение cross-contamination).
- **Длительность:** Расчёт через калькулятор мощности теста (Power=80%, α=0.05, MDE=5%). Ориентировочно: 14 дней или до набора ~10 000 заявок в каждой группе.
- **Контроль внешних факторов:** Фиксация сезонности, маркетинговых акций и изменений в продукте на период теста.

## 3. Метрики оценки
| Тип | Метрика | Обоснование |
|-----|---------|-------------|
| **Основная** | `F1-score` (класс 1) | Баланс между Precision и Recall. В скоринге важно не пропустить дефолтника (Recall) и не отклонить платёжеспособного клиента (Precision). |
| **Дополнительная** | `Precision` | Минимизация False Positive. Ошибка I рода ведёт к упущенной прибыли (отказ надёжному клиенту). |
| **Бизнес-метрика** | `Доля просрочек > 90 дней` | Прямое влияние на риск-аппетит банка и provisioning. |

## 4. Статистический анализ
1. **Кумулятивные метрики:** Отслеживание F1/Precision в динамике по дням. Остановка анализа только после стабилизации кривых (решение проблемы *peeking problem*).
2. **Статистический тест:** `Z-тест для пропорций` (`statsmodels.stats.proportion.proportions_ztest`) для сравнения долей ошибок/успехов между группами. Альтернатива при малых выборках: точный критерий Фишера.
3. **Доверительные интервалы:** Построение 95% ДИ для разницы метрик (`v2 - v1`). Если интервал не пересекает 0 и лежит в положительной области → разница статистически значима.
4. **Критерий успеха:** `p-value < 0.05`, нижняя граница ДИ > 0, рост F1 ≥ MDE, отсутствие деградации Precision.

## 5. Практическая демонстрация в API
Для валидации плана в рамках проекта реализован переключатель версий через ENV-переменную или query-параметр:
```python
# app/main.py (фрагмент для демонстрации)
MODEL_VERSION = os.getenv("MODEL_VERSION", "v1")  # v1 или v2

@app.route('/predict', methods=['POST'])
def predict():
    # ... валидация данных ...
    if MODEL_VERSION == "v2":
        prediction = pipeline_v2.predict(df_input)[0]
        proba = pipeline_v2.predict_proba(df_input)[0][1]
    else:
        prediction = pipeline_v1.predict(df_input)[0]
        proba = pipeline_v1.predict_proba(df_input)[0][1]
    return jsonify({"prediction": int(prediction), "probability": float(proba), "model_version": MODEL_VERSION})
```
